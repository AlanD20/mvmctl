# Sudo Privilege Architecture: Group-Based Passwordless Access

**Status:** accepted
**Date:** 2026-05-22

The project uses a **tightly scoped, group-based sudo model** where members of a dedicated Unix group (`mvm`) can execute specific privileged commands via `sudo` without a password prompt. This avoids running the entire CLI as root while still allowing VM creation (loop-mount), network management (ip/iptables/nft), kernel module loading (modprobe), and sysctl configuration.

## The Group Activation Mechanism

`sg mvm` (switch group) changes the **primary GID** of the current process to the `mvm` group. This is visible in `id` output:

```
$ sg mvm -c 'id'
uid=1000(www) gid=957(mvm) groups=957(mvm),98(power),961(docker),...,1001(www)
```

Key implications:
- `os.Getuid()` returns 1000 (not root), so `os.Getuid() != 0` is true and `sudo` IS prepended
- `os.Getgroups()` includes GID 957, so `CheckPrivileges()` passes
- The primary GID changes from the user's normal group to mvm
- The supplementary group list preserves membership in ALL other groups

## The Actual Sudoers File

`mvm host init` creates `/etc/sudoers.d/mvm`. The file contains:

```
%mvm ALL=(root) NOPASSWD: /usr/sbin/ip, /usr/sbin/iptables, /usr/sbin/iptables-restore, /usr/sbin/iptables-save, /usr/sbin/nft, /usr/sbin/sysctl, /usr/sbin/modprobe, /home/www/.cache/mvmctl/bin/mvm
```

The sudoers file is generated by `HostService.GenerateSudoersContent()` using two categories from `internal/infra/constants.go`:

- **`PrivilegedBinaries`** — system binaries with hardcoded absolute paths:
  - `/usr/sbin/ip` (iproute2)
  - `/usr/sbin/iptables` (iptables)
  - `/usr/sbin/iptables-restore` (iptables)
  - `/usr/sbin/iptables-save` (iptables)
  - `/usr/sbin/nft` (nftables)
  - `/usr/sbin/sysctl` (procps)
  - `/usr/sbin/modprobe` (kmod)

- **Service binaries** — the `mvm` binary itself for the `mvm run provision` subcommand:
  - `mvm` → resolves to the actual binary path

NOTHING else is permitted. `sudo -n true`, `sudo cat /etc/sudoers.d/mvm`, `sudo test -f ...`, and any other non-listed command WILL prompt for a password. This is by design and is NOT a configuration problem.

## How VM Creation Works Without losetup in Sudoers

This is the critical design insight: **all loop-mount operations are delegated to the `mvm run provision` service subprocess**. The call chain is:

```
mvm vm create
  → ImageProvisioner uses loop-mount backend (not guestfs)
  → system.RunCmd(privileged=true) is called with args=["sudo", "mvm", "run", "provision", ...]
       ↑ CheckPrivileges() passes
       ↑ sudoers allows %mvm to run mvm without password
       ↑ mvm is now running as root (for the provision subcommand)
          → Inside provision: losetup, mount, umount, etc. — all as root
```

This means:
- The sudoers file does NOT need `losetup`, `mount`, `umount`, `mkfs.ext4`, `blkid`, or any filesystem tool
- ALL filesystem operations are encapsulated in the compiled `mvm` binary's `run provision` subcommand
- The `mvm` binary is the single sudoers entry point for VM creation

## How System Tests Interact with Sudo

When running `sg mvm -c 'pytest tests/e2e/'`:

1. `sg mvm` sets the process primary GID to `mvm` (957)
2. The pytest process inherits `mvm` as its primary group
3. All subprocesses spawned by tests inherit the `mvm` primary group
4. When `mvm vm create` calls `system.RunCmd(privileged=true)`:
   - `CheckPrivileges()` checks: primary GID == mvm GID ✅, user in group members ✅, GID in process groups ✅
   - `sudo` is prepended to the command
   - The sudoers drop-in grants NOPASSWD access
   - The command executes as root without any password prompt

## Why Not Run Everything as Root

- **Principle of least privilege.** The CLI runs as a normal user for HTTP downloads, file I/O, SQLite queries, and all non-privileged operations. Only specific system binaries are elevated.
- **Safety and auditability.** The sudoers file is a compact, auditable list of exactly what can run as root. A compromise of the CLI does not automatically grant root access.
- **Development ergonomics.** Developers build and iterate as a normal user. Only one-time setup (`mvm host init`) needs `sudo`.

## Why a Group-Based Model

- The `mvm` group cleanly separates "can manage VMs" from "can't manage VMs" at the OS level
- `sg mvm` activates the group for the current shell session without requiring logout/login, CI restart, or environment manipulation
- System tests run under `sg mvm -c 'pytest ...'`, inheriting the group across all subprocesses at zero cost
- The setup is **one command** executed once: `sudo mvm host init`

## Consequences

- `sudo -n true`, `sudo -v`, and any non-listed command will fail for `mvm` group members. The `CheckPrivileges()` function is aware of this and does NOT gate privileged execution on credential validation.
- Adding a new privileged binary requires updating `internal/infra/constants.go` (add to `PrivilegedBinaries`) AND regenerating the sudoers file via `mvm host init`.
- The `mvm` binary MUST exist at the path listed in the sudoers file for VM creation to work. If it's missing — as can happen after `cache clean --force` — re-extract via `mvm cache init`.
- The group name is the CLI name (`infra.MVMUnixGroup = infra.CLIName`). Currently `"mvm"`, so the group is `mvm`.
- `sg mvm` is a prerequisite for VM-related system tests. Non-VM tests work without it.
- Tests that involve VM creation MUST verify the `mvm` group is active before attempting to create VMs.
