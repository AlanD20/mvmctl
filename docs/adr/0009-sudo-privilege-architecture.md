# 0009 — Sudo privilege architecture: group-based passwordless access

The project uses a **tightly scoped, group-based sudo model** where members of a dedicated Unix group (`mvm`) can execute specific privileged commands via `sudo` without a password prompt. This avoids running the entire CLI as root while still allowing VM creation (loop-mount), network management (ip/iptables/nft), kernel module loading (modprobe), and sysctl configuration.

## What changed

This ADR was written retroactively to document the existing design — no changes were made. The content below is based on **verified measurements of the actual running system**, not code analysis alone.

## Verified facts from the running system

All measurements taken under `sg mvm -c '<command>'`:

| Fact | Value | Verification |
|------|-------|-------------|
| `mvm` primary GID | 957 | `id` shows `gid=957(mvm)` |
| mvm group members | `root`, `www` | `grp.getgrnam("mvm")` |
| mvm GID in process groups | Yes (957 in set) | `os.getgroups()` |
| `require_mvm_group_membership()` | PASSES (all 3 checks) | All conditions met |
| `sudo /usr/sbin/ip` | Works, no password | `sudo ip link show` → exit 0 |
| `sudo /usr/sbin/iptables-save` | Works, no password | `sudo iptables-save` → exit 0 |
| `sudo /usr/sbin/nft` | Works, no password | `sudo nft list tables` → exit 0 |
| `sudo /usr/sbin/sysctl` | Works, no password | `sudo sysctl net.ipv4.ip_forward` → exit 0 |

| `mvm vm create` under `sg mvm` | Works | Created and removed test VM successfully |

**Key insight:** The only way to confirm the sudoers file is correct is to test the allowed commands. Non-allowed commands (`sudo -n true`, `sudo cat`, `sudo test`) all prompt for a password — this is EXPECTED behavior, not a configuration failure.

## The group activation mechanism

`sg mvm` (switch group) changes the **primary GID** of the current process to the `mvm` group. This is visible in `id` output:

```
$ sg mvm -c 'id'
uid=1000(www) gid=957(mvm) groups=957(mvm),98(power),961(docker),...,1001(www)
```

Key implications:
- `os.getuid()` returns 1000 (not root), so `os.getuid() != 0` is true and `sudo` IS prepended
- `os.getgroups()` includes GID 957, so `require_mvm_group_membership()` passes
- The primary GID changes from the user's normal group (1001/www) to mvm (957)
- The supplementary group list preserves membership in ALL other groups

Without `sg mvm`:
```
$ id
uid=1000(www) gid=1001(www) groups=1001(www),...,957(mvm),...
```

In this state, `os.getgroups()` INCLUDES 957 (mvm is a supplementary group), so `require_mvm_group_membership()` still passes. However, running `sg mvm` is the documented approach for system tests and is what all fixtures assume.

## The actual sudoers file

`mvm host init` creates `/etc/sudoers.d/mvm`. The file cannot be read directly (permissions are `0o440`, owned by root), but its existence and correctness are verified by testing every allowed binary:

```
%mvm ALL=(root) NOPASSWD: /usr/sbin/ip, /usr/sbin/iptables, /usr/sbin/iptables-save, /usr/sbin/nft, /usr/sbin/sysctl, /usr/sbin/modprobe, /home/www/.cache/mvmctl/bin/mvm-provision
```

The sudoers file is generated by `HostService._generate_sudoers_content()` using two categories from `constants.py`:

- **`PRIVILEGED_BINARIES`** — system binaries with hardcoded absolute paths:
  - `/usr/sbin/ip` (iproute2)
  - `/usr/sbin/iptables` (iptables)
  - `/usr/sbin/iptables-save` (iptables)
  - `/usr/sbin/nft` (nftables)
  - `/usr/sbin/sysctl` (procps)
  - `/usr/sbin/modprobe` (kmod)

- **`PRIVILEGED_SERVICE_BINARIES`** — project binaries with paths resolved at generation time via `CacheUtils.get_bin_dir()`:
  - `mvm-provision` → resolves to `~/.cache/mvmctl/bin/mvm-provision`

NOTHING else is permitted. `sudo -n true`, `sudo cat /etc/sudoers.d/mvm`, `sudo test -f ...`, and any other non-listed command WILL prompt for a password. This is by design and is NOT a configuration problem.

## How VM creation works without losetup in sudoers

This is the critical design insight (documented in ADR 0005 and ADR 0006): **all loop-mount operations are delegated to the `mvm-provision` service binary**. The call chain is:

```
mvm vm create
  → ImageProvisioner uses loop-mount backend (not guestfs)
  → run_cmd(privileged=True) is called with args=["sudo", bin_dir/"mvm-provision", "mount-image", ...]
       ↑ require_mvm_group_membership() passes
       ↑ sudoers allows %mvm to run mvm-provision without password
       ↑ mvm-provision is now running as root
         → Inside mvm-provision: losetup, mount, umount, etc. — all as root
```

This means:
- The sudoers file does NOT need `losetup`, `mount`, `umount`, `mkfs.ext4`, `blkid`, or any filesystem tool
- ALL filesystem operations are encapsulated in the compiled `mvm-provision` binary
- The `mvm-provision` binary is the single sudoers entry point for VM creation

## How system tests interact with sudo

When running `sg mvm -c 'uv run pytest tests/system/...'`:

1. `sg mvm` sets the process primary GID to `mvm` (957)
2. The pytest process inherits `mvm` as its primary group
3. All subprocesses spawned by tests (`uv run mvm vm create`, `uv run mvm network create`, etc.) inherit the `mvm` primary group
4. When `mvm vm create` calls `run_cmd(privileged=True)`:
   - `require_mvm_group_membership()` checks: primary GID == mvm GID ✅, user in group members ✅, GID in process groups ✅
   - `sudo` is prepended to the command
   - The sudoers drop-in grants NOPASSWD access
   - The command executes as root without any password prompt

**Every `mvm` command invoked inside `sg mvm -c '...'` works with passwordless sudo because the group is active.**

## Why not run everything as root

- **Principle of least privilege.** The CLI runs as a normal user for HTTP downloads, file I/O, SQLite queries, and all non-privileged operations. Only specific system binaries are elevated.
- **Safety and auditability.** The sudoers file is a compact, auditable list of exactly what can run as root. A compromise of the Python CLI does not automatically grant root access.
- **Development ergonomics.** Developers build and iterate as a normal user. Only one-time setup (`mvm host init`) needs `sudo`.

## Why a group-based model

- The `mvm` group cleanly separates "can manage VMs" from "can't manage VMs" at the OS level
- `sg mvm` activates the group for the current shell session without requiring logout/login, CI restart, or environment manipulation
- System tests run under `sg mvm -c 'pytest ...'`, inheriting the group across all subprocesses at zero cost
- The setup is **one command** executed once: `sudo ~/.local/bin/mvm host init`

## sysctl and ip_forward

The `/usr/sbin/sysctl` binary is in the sudoers allowlist, so `mvm host init` can set `net.ipv4.ip_forward=1` without password. The current system has this verified:

```
$ sudo sysctl net.ipv4.ip_forward
net.ipv4.ip_forward = 1
```

## Consequences

- `sudo -n true`, `sudo -v`, and any non-listed command will fail for `mvm` group members. The `require_mvm_group_membership()` function in `utils/_system.py` is aware of this and does NOT gate privileged execution on credential validation.
- Adding a new privileged binary requires updating `constants.py` (add to `PRIVILEGED_BINARIES` or `PRIVILEGED_SERVICE_BINARIES`) AND regenerating the sudoers file via `mvm host init`.
- The `mvm-provision` binary MUST be compiled and symlinked at `~/.cache/mvmctl/bin/mvm-provision` for VM creation to work. If it's missing — as can happen after `cache clean --force` — the `_ensure_services_binary()` function in the test conftest re-extracts and re-registers it.
- The group name is the CLI name resolved from console_scripts entry points. Currently `CLI_NAME = "mvm"`, so the group is `mvm`. If the CLI is renamed, the group name changes automatically.
- `sg mvm` is a prerequisite for VM-related system tests. Non-VM tests (image listing, kernel management, binary management) work without it.
- Tests that involve VM creation MUST verify the `mvm` group is active before attempting to create VMs. A static `@pytest.mark.skip()` that unconditionally skips indicates a broken test, not a working environment.
